# AlgoMind 前后端对接开发文档

## 1. 文档目标

本版本文档用于前后端联调，聚焦以下内容：
- 每个接口请求时前端会抛出什么（请求参数、Header、异常处理）。
- 每个接口后端应返回什么（统一响应结构、字段类型、业务码）。
- 响应数据结构（列表、详情、分页、判题结果、AI分析结果等）。

说明：
- 本项目当前前端包含本地兜底逻辑（localStorage + mock）。
- 为了稳定联调，建议后端优先满足本文档的字段结构；前端即可无痛切换真实接口。

---

## 2. 基础约定

### 2.1 Base URL

前端 Axios 配置位于 `src/api/index.js`：
- `baseURL = import.meta.env.VITE_API_BASE_URL || '/api'`
- 默认请求路径都在 `/api` 下。

### 2.2 超时与认证

- 超时：`10000ms`
- 前端请求拦截器会从 `localStorage.user` 读取 `token`，并自动注入：
  - `Authorization: Bearer <token>`

后端要求：
- 需要鉴权的接口校验 `Authorization`。
- 无 token 或 token 无效时，返回统一错误结构（见 2.4）。

### 2.3 Content-Type

- 默认：`application/json`

### 2.4 统一响应结构（强约束）

前端当前按如下结构判断成功：`res.data.code === 0`。

成功响应：

```json
{
  "code": 0,
  "message": "ok",
  "data": {}
}
```

失败响应：

```json
{
  "code": 40001,
  "message": "参数错误",
  "data": null
}
```

推荐业务码：
- `0`：成功
- `40001`：参数错误
- `40003`：未登录/Token失效
- `40004`：无权限
- `40400`：资源不存在
- `40900`：资源冲突（如重复提交）
- `50000`：服务内部错误

### 2.5 错误抛出约定（前端会抛出什么）

前端 Axios 响应拦截器对非 2xx 或网络错误执行 `Promise.reject(error)`。

后端注意：
- 若返回 4xx/5xx，前端会进入 `catch`，并触发页面兜底逻辑（本地数据）。
- 若希望前端进入成功分支，必须 HTTP 200 且 `code = 0`。

前端捕获到的常见异常类型：
- 网络异常（断网、跨域、超时）
- HTTP 状态异常（401/403/404/500）
- 业务失败（HTTP 200 但 `code !== 0`）

## 4.1 登录

- 方法：`POST`
- 路径：`/api/login`
- 调用位置：`src/stores/user.js`

请求体：

```json
{
  "username": "guest",
  "password": "guest"
}
```

成功响应：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "user": {
      "id": 1,
      "name": "游客",
      "token": "jwt-token",
      "bio": "先把基础打扎实，再冲更高难度。",
      "gender": "unknown",
      "avatar": "",
      "targetTrack": "algo",
      "weeklyGoal": 10
    },
    "points": 120
  }
}
```

失败响应：

```json
{
  "code": 40001,
  "message": "用户名或密码错误",
  "data": null
}
```

前端抛出/处理：
- `code !== 0` 或请求异常时，登录函数会返回 `false` 或进入本地模拟登录兜底。

## 4.3 提交作答/代码评测

- 方法：`POST`
- 路径：`/api/submit`
- 调用位置：`src/stores/level.js`, `src/views/Challenge.vue`

请求体：

```json
{
  "levelId": 5,
  "answer": "用户答案或代码文本",
  "meta": {
    "attempts": 1,
    "timeMs": 42000,
    "language": "cpp",
    "stdinInput": "1 2 3"
  },
  "language": "cpp",
  "stdinInput": "1 2 3",
  "code": "int main() { return 0; }"
}
```

说明：
- 前端会同时传 `answer` 和（当 answer 是字符串时）`code`。
- 后端可优先使用 `code` 做代码评测。

成功响应（通用）：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "correct": true,
    "pointsEarned": 18,
    "nextLevelUnlocked": true,
    "starsEarned": 3,
    "bestStars": 3,
    "timeMs": 42000,
    "score": 100,
    "output": "程序标准输出",
    "analysis": "AI分析文本"
  }
}
```

字段要求：
- 客观题主要使用：`correct`, `pointsEarned`, `nextLevelUnlocked`, `starsEarned`, `bestStars`
- 代码题评估面板使用：`score`, `output`, `analysis`, `pointsEarned`

失败响应：

```json
{
  "code": 40001,
  "message": "提交参数不完整",
  "data": null
}
```

前端抛出/处理：
- 接口异常时降级为本地判题。
- 代码题若返回缺失字段，会显示默认值（如暂无输出）。

## 5. 兼容性说明（重点）

当前项目存在两套返回判断方式：
- Store 大部分接口：`code === 0`
- Profile 页接口：`success === true`

后端联调建议（短期最稳）：
- 所有接口都返回：`code` + `success` + `message` + `data`

示例：

```json
{
  "code": 0,
  "success": true,
  "message": "ok",
  "data": {}
}
```

这样前端无需改动即可覆盖全部页面。

## 6. 典型错误场景与抛出内容

### 6.1 Token 失效

后端响应：

```json
{
  "code": 40003,
  "success": false,
  "message": "登录已过期",
  "data": null
}
```

HTTP 状态可用 `401`。

前端表现：
- 当前不会自动跳登录页；会进入 catch 或业务失败分支。
- 建议后续加统一 401 处理。

### 6.2 参数校验失败

后端响应：

```json
{
  "code": 40001,
  "success": false,
  "message": "levelId不能为空",
  "data": null
}
```

前端表现：
- 关卡提交失败时提示“提交失败，请稍后重试”或走本地判题。

### 6.3 服务异常

后端响应：

```json
{
  "code": 50000,
  "success": false,
  "message": "服务内部错误",
  "data": null
}
```

前端表现：
- 论坛/错题/关卡等模块自动回退本地数据。

## 7. 联调建议

- 第一阶段：先打通 8 个核心接口（登录、关卡列表、提交、错题列表/新增/分析、帖子列表/新增）。
- 第二阶段：补齐个人中心 4 个接口（热力图、统计、活动、资料更新）。
- 第三阶段：统一返回格式，减少页面兼容分支。

推荐后端自测清单：
- 所有接口都验证 `Authorization` 的兼容情况。
- 所有成功响应都含 `data` 字段，且类型固定。
- 所有失败响应都含 `code/message`，便于前端直接提示。
